<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → FastAPI route | Untrusted user input (document IDs, answers, chat messages) |
| Route → Quota service | Trusted internal call, but quota enforcement must happen before chain execution |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | enforce_quota dependency | mitigate | Dependency uses `get_current_user` which validates Clerk JWT; quota is tied to authenticated user |
| T-02-02 | Tampering | quota_service increment | mitigate | Increment is transactional via SQLAlchemy update; quota check occurs before increment |
| T-02-03 | Repudiation | quota usage tracking | accept | No audit logs required for MVP; quota counts are sufficient |
| T-02-04 | Information Disclosure | quota limits in code | accept | Limits are not secret; exposed in open source |
| T-02-05 | Denial of Service | quota exhaustion | mitigate | Daily resets prevent permanent exhaustion; Pro tier has higher limits |
| T-02-06 | Elevation of Privilege | tier checking in quota_service | mitigate | Quota service reads `user.tier` from database, not from user input |
</threat_model>
