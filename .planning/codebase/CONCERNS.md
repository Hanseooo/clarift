## Security Considerations

### No Environment File Templates
- **Risk:** No `.env.example` file exists in either `backend/` or `frontend/`. The architecture doc (`docs/dev/architecture.md`) lists 14 environment variables including secrets (`NEXTAUTH_SECRET`, `JWT_SECRET`, `GEMINI_API_KEY`, `R2_SECRET_ACCESS_KEY`, `PAYMONGO_SECRET_KEY`). Without a template, developers may hardcode values or misconfigure secrets.
- **Files:** `backend/` (no `.env.example`), `frontend/` (no `.env.example`)
- **Current mitigation:** `.gitignore` present in both directories.
- **Recommendations:** Create `.env.example` files in both `backend/` and `frontend/` with all required variable names and placeholder values. Document which vars must be identical across services (`NEXTAUTH_SECRET` = `JWT_SECRET`).

### JWT Secret Alignment Risk
- **Risk:** `NEXTAUTH_SECRET` (Next.js) and `JWT_SECRET` (FastAPI) must be identical for auth to work. If they drift, all authenticated requests fail silently or with opaque errors.
- **Files:** `docs/dev/architecture.md` (line 97), deployment configuration (not yet present)
- **Current mitigation:** Documented requirement only.
- **Recommendations:** Add a startup check in FastAPI that validates the JWT secret can decode a test token. Add CI check that both services receive the same secret in deployment.

### No Auth Implementation Yet
- **Risk:** No NextAuth configuration, no JWT verification, no auth middleware, no protected routes. The entire auth flow documented in `docs/dev/architecture.md` is unimplemented.
- **Files:** `frontend/src/app/` (no auth routes), `backend/` (no auth routes)
- **Current mitigation:** None â€” the app has no authentication surface yet.
- **Recommendations:** Implement auth first per `docs/dev/implementation-plan.md` priority (Days 3-4). Never build features that require user context before auth exists.

### Vector Query User-Scoping Not Implemented
- **Risk:** The architecture mandates `WHERE user_id = :user_id` on all pgvector similarity searches. This is a security requirement â€” without it, users could retrieve chunks from other users' documents.
- **Files:** `backend/` (no chains or services exist yet)
- **Current mitigation:** Documented as non-negotiable rule in `docs/dev/agents.md` (rule 3) and `docs/dev/decisions.md`.
- **Recommendations:** When chains are implemented, add a test specifically verifying that vector queries include user_id filtering. Never merge a chain without this test.

## Fragile Areas

### Dual Schema Ownership (Drizzle + SQLAlchemy)
- **Files:** `frontend/src/db/` (not yet created), `backend/app/db/models.py` (not yet created)
- **Why fragile:** Two ORMs describe the same 10-table schema. If Alembic migration and Drizzle schema drift, queries will fail at runtime with cryptic column-not-found errors.
- **Safe modification:** Always update both in the same commit. Alembic migration first, then Drizzle schema. Verify with type-check after Drizzle update.
- **Test coverage:** No tests exist for schema alignment. Add a smoke test that verifies both ORMs can read/write the same table.

### SSE Job Status Pattern
- **Files:** Not implemented (documented in `docs/dev/architecture.md`)
- **Why fragile:** SSE connections can drop silently. Client must handle reconnection, partial state recovery, and stale job IDs. MSW (frontend test mock) doesn't support SSE natively per `docs/dev/testing-strategy.md`.
- **Safe modification:** Implement SSE with explicit reconnection logic. Add timeout guards. Test SSE separately from MSW mocks.
- **Test coverage:** No SSE tests exist. Follow pattern in `docs/dev/testing-strategy.md` for SSE endpoint testing.

### ARQ Worker + FastAPI Shared Codebase
- **Files:** `backend/` (no worker code)
- **Why fragile:** ARQ worker and FastAPI web process share the same codebase but run as separate Railway processes. Shared state, DB sessions, or Redis connections can cause cross-process interference.
- **Safe modification:** Keep worker and web process stateless relative to each other. Use Redis queue as the only communication channel. Never share in-memory state.
- **Test coverage:** No worker tests exist. Follow `docs/dev/testing-strategy.md` ARQ worker testing pattern with mocked dependencies.

## Scaling Limits

### Neon PostgreSQL Connection Limits
- **Current capacity:** Neon free tier: 90 max connections.
- **Limit:** Next.js serverless functions + FastAPI web + ARQ worker all consume connections. Serverless cold starts can exhaust pool.
- **Scaling path:** Use Neon's built-in connection pooler. Set pool size appropriately. Monitor connection count in Neon dashboard.

### Gemini API Rate Limits
- **Current capacity:** Gemini 2.5 Flash has rate limits (tokens per minute, requests per minute).
- **Limit:** Concurrent document processing + summary + quiz + practice generation can hit rate limits quickly.
- **Scaling path:** Tenacity retry with exponential backoff (documented in `docs/dev/decisions.md`). ARQ queue serializes heavy operations. Monitor token usage in Sentry.
